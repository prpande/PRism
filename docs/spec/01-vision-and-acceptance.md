# Vision & Acceptance

## What we're building

A local web application that helps a single reviewer review GitHub pull requests with an experience that is **deliberately better than GitHub's native review UI** for the things reviewers actually do every day. The PoC ships zero AI features but architects clean seams so that v2 can layer Claude-powered augmentation on top without refactoring core code.

The reviewer's machine runs the app. The browser is the UI. GitHub is the source of truth. The tool's value is in the experience between those two endpoints.

## Why this exists

GitHub's review UI works but is not designed for the reviewer's *focus and flow*. As more code is generated by AI, the bottleneck shifts from authoring to reviewing — and the reviewing experience needs to grow up. This tool exists to:

1. Make day-to-day code review feel deliberate, fast, and pleasant — not like an obstacle course.
2. Provide a clean substrate for AI augmentation in v2 — one that exposes review state and intent to LLMs in a structured way.
3. Stay local-first and reviewer-first. No team SaaS, no telemetry, no lock-in.

### A note on the trajectory

The framing above assumes humans are the primary reviewers and AI is a tool they reach for. There is a credible alternate trajectory in which **AI also reviews PRs before humans see them**, and the human's role shifts toward "review the AI's review" and approve aggregations. Tools like Copilot Workspace and emerging review-bots are already moving in this direction in 2026.

If that trajectory dominates within 12–24 months, a tool optimized for human-led review with AI-on-the-side may be designed for a transitional period. The seams in `04-ai-seam-architecture.md` partly hedge against this — a `IPreSubmitValidator` registered with an "AI reviewer" implementation can act as the first reviewer, and the human becomes the second — but the *primary surface* in this spec is still a human reading a diff. We accept the bet that this primary surface remains valuable for years, while leaving the AI-first review evolution path open via the existing seams. This is documented here so the next maintainer knows it was considered, not overlooked.

## The wedge — what makes this tool worth opening instead of GitHub.com

These are the concrete differentiators the PoC delivers. Each is a deliberate response to a felt pain in the GitHub experience. **The wedge leads with what github.com does not do at all** (stale-draft reconciliation, iteration tabs); the supporting items are local-first invariants that protect the reviewer's work even when github.com offers similar surfaces.

- **Stale-draft reconciliation** *(moment of control; not headline-marketable on its own)*. When commits arrive, every draft comment gets an automatic fresh / moved / stale classification with explicit per-draft reconciliation. The reviewer adjudicates before submit. No silent loss of work, no comments landing on the wrong line. github.com auto-reanchors or invalidates drafts server-side without input; PRism makes the user the deciding party. (See `spec/03-poc-features.md` § 5 for the seven-row classification.) *Honest framing*: this feature is a real difference from github.com, but the user friction it defends against (drafts being silently re-anchored on a force-push) is **rare in practice**. Most "draft loss" reviewers experience is browser-tab loss — solved by *any* local persistence, not by the seven-row matrix. The reconciliation matters when it matters, but does not, on its own, motivate switching from github.com. The headline of the wedge is the *combination* below; calling reconciliation the single headline differentiator was earlier framing this spec has since stepped back from.
- **Iteration tabs.** First-class navigation through each push as a discrete iteration with its own diff range, plus an arbitrary-pair "Compare ⇄" picker. github.com does not surface iterations as navigable tabs.
- **File-by-file diff view** with always-visible file tree and `j`/`k` navigation. No infinite scroll through a 50-file PR. Modeled on CodeFlow / Azure DevOps / VS Code's PR extension.
- **Local-first authoring with reviewer-atomic submit** *(supporting invariant, not the headline)*. Drafts persist on disk across sessions, network hiccups, GitHub outages — github.com's drafts live server-side and can be lost on unexpected navigation. Drafts, replies, verdict, and summary attach to a GitHub *pending review* and finalize together when the reviewer clicks Submit. The pending-review pipeline itself is the same one github.com's "Start a review → Finish your review" flow uses; what makes PRism different is *what happens to drafts that aren't yet submitted*. (See [verification-notes § C1](./00-verification-notes.md#c1) for the technical grounding.)
- **Banner-based update model.** When a teammate pushes mid-review, a non-intrusive banner appears. The diff under the reviewer's cursor never mutates without consent.
- **Rendered markdown for `.md` files** by default (with a Diff toggle), including Mermaid diagrams. GitHub forces reviewers to read markdown-as-code; we don't.
- **Word-level diff** within changed lines. Whitespace changes are shown as-is — v2's AI categorizes; PoC is truthful.
- **Inbox with five sections** matching real reviewer behavior: review-requested, awaiting-author, authored-by-me, mentioned, CI-failing-on-my-PRs. With unread badges showing per-PR deltas since last visit.
- **Reply to existing GitHub comments** as part of the submitted review. Replies attach to the same pending review and finalize when the user submits.
- **Designed AI seams everywhere.** Every UI surface that v2 will want to enrich already has a capability-flag-gated slot. Every state mutation that v2 might want to react to flows through an event bus. Core types do not need to be reshaped to add AI features.

## Core principles

These principles override any specific feature decision. When a future contributor wonders "should we do X?", these are the tiebreakers:

1. **Truthful by default, clever in v2.** PoC shows everything as it actually is — including whitespace changes, unfiltered diffs, all draft comments. Filtering and categorization is the AI layer's job. We don't build in opinions about what the reviewer "shouldn't have to see."
2. **The reviewer's text is sacred.** AI suggestions never auto-apply. Drafts are never silently dropped. The user explicitly chooses every state change.
3. **Banner, not mutation.** Remote state never auto-applies *to the code under review or to the reviewer's drafts*. The diff under the reviewer's cursor never changes without an explicit reload action; drafts never silently re-anchor. **One narrow exception:** existing comments rendered inline as widgets between code lines update on poll if their text changed on github.com, and threading-state badges (resolved / unresolved when v2 ships P4-A1) update on poll. These are informational signals about *other people's content*, not about the code or the reviewer's own work; the principle applies strictly to the code and the drafts. Local user actions (saving a draft, marking viewed, opening a composer) trigger UI mutations as expected — this principle constrains *remote-driven* changes only.
4. **Reviewer-atomic submit.** Drafts and replies stage in a GitHub pending review (invisible to others); the user's submit click finalizes them together. Until submit, nothing is visible publicly. Local drafts are also retained until the submit succeeds.
5. **Local-first, third-party calls are explicit and opt-in.** No telemetry, no analytics. The user's GitHub PAT and (in v2) LLM credentials live in the OS keychain. **AI features in v2 send PR content (diffs, titles, descriptions, comments) to the configured LLM provider** — that is a third-party data flow, not telemetry, and it is opt-in per feature. The principle reduces to: the tool never sends data anywhere the user hasn't explicitly authorized for a specific feature.
6. **GitHub-coupled, internally clean.** PoC commits to GitHub as the only backend (github.com cloud and GitHub Enterprise Server, both supported via a configurable host). There is no multi-provider abstraction — no `IReviewProvider` interface that promises pluggability, no Azure DevOps / GitLab / Bitbucket adapters in the roadmap, no extensibility bags for provider-specific verdicts. GitHub-specific code lives in its own project (`PRism.GitHub`) for clean dependencies and testability, but the project is *the* implementation, not "the GitHub adapter." If a non-GitHub backend is ever wanted, it would be a v2+ refactor, not a feature of this architecture.
7. **Capability-flag-gated AI surfaces.** Every AI slot in the UI is hidden in PoC via a single `/api/capabilities` check. v2 lights them up by registering implementations and flipping flags.

## The PoC demo (litmus test)

If the PoC can do exactly the following end-to-end, it is done:

1. Launch the binary → browser opens to `http://localhost:<port>`
2. First-run setup screen → user pastes a fine-grained GitHub PAT
3. Inbox loads with five sections; PRs across all the user's repos are visible, with unread badges where appropriate
4. User clicks a PR row → PR detail view loads with header, file tree, side-by-side diff (default), iteration tabs
5. User navigates files with `j`/`k`, marks files as viewed
6. User clicks an iteration tab → sees what changed in that iteration vs the prior one
7. User clicks a line in the diff → comment composer opens; user writes a markdown comment; saves draft
8. User opens a `.md` file → sees rendered markdown by default (with Mermaid diagrams rendered), can toggle to Diff view
9. While the user is reviewing, a teammate pushes new commits. A banner appears: *"PR updated — Reload"*
10. User clicks Reload → all draft comments are auto-classified fresh / moved / stale; verdict (if set) is flagged for re-confirmation
11. User reconciles each stale item, sets a verdict, writes a PR-level summary
12. User clicks "Submit review" → backend finalizes the pending review on GitHub via the GraphQL pending-review pipeline (verdict + summary + new threads + replies). The pipeline is the same one github.com's "Start a review → Finish your review" flow uses — what's specific to PRism is the local-first authoring and explicit reconciliation that precede this step.
13. The review appears on GitHub correctly anchored to the right lines

**No AI features run during this flow.** The AI seams exist in the code but every capability flag returns `false` and the corresponding UI slots render `null`.

## Definition of done for the PoC

The PoC is complete when **all** of the following are true:

### Functional
- [ ] The demo flow above passes end-to-end on Windows and macOS (Apple Silicon).
- [ ] The "Paste PR URL" jump-to escape hatch works from the inbox.
- [ ] Draft state survives application restart (drafts persist across sessions).
- [ ] All five inbox sections populate correctly, including "CI failing on my PRs" via the GitHub Checks API.
- [ ] Per-iteration tab content shows the correct diff range (`iter_N-1_head..iter_N_head`).
- [ ] The arbitrary-pair "Compare ⇄" picker correctly diffs any two iterations.
- [ ] Word-level diff highlighting works on changed lines.
- [ ] Whitespace changes are rendered (no filtering).
- [ ] Mermaid diagrams render in `.md` files in the rendered view.
- [ ] Markdown bodies in comments render with GFM features (tables, task lists, strikethrough).
- [ ] Replies to existing GitHub comments attach to the pending review and appear correctly threaded on GitHub after submit. (See [verification-notes § C1](./00-verification-notes.md#c1).)
- [ ] Stale-draft reconciliation classifies correctly under every branch documented in `03-poc-features.md` § 5: line content unchanged at original line; same content at a different line (single match); same content at multiple lines (closest-line tie-break); whitespace-equivalent match (auto-formatter ran or CRLF↔LF flipped); file renamed via `renamed` status; file deleted; no match anywhere (Stale).
- [ ] Submit blocks when any draft is in `stale` status until reconciled.
- [ ] Verdict re-confirmation is required after any new commit. Submit is blocked when the banner-detected `head_sha` differs from what the user has reloaded against — the user must click Reload (which runs the reconciliation pass and flips `draftVerdictStatus` to `needs-reconfirm` if a verdict was set) before Submit re-enables. This closes the "Approve without seeing the new commit" footgun (per `03-poc-features.md` § 6 Submit rule (f)).
- [ ] Banner-based polling refreshes the active PR every 30s and the inbox every 2 minutes (configurable).
- [ ] Submit failure preserves all drafts and `pendingReviewId` in `state.json`, shows a clear error message, and retry from the same state converges on success **without producing duplicate threads or replies on GitHub** (this is unconditional — duplicate-thread outcomes are not an acceptable recovery shape). Includes the **lost-response window** (response from a successful `addPullRequestReviewThread` never reaches the local store) — retry's pre-reconciliation step adopts the orphan server thread by parsing the `<!-- prism:client-id:<draft.id> -->` HTML-comment marker out of the returned body and matching against the unstamped draft's `id`. Verified by an integration test that simulates a mid-mutation network drop after the server-side mutation has succeeded; the test asserts that the post-retry pending review carries exactly one thread per draft, no duplicates. See [verification-notes § C7](./00-verification-notes.md#c7) for the empirical gate on marker durability and the documented fallback if the marker is stripped.
- [ ] Submit fails, the PR is then closed by the author before the user retries: `state.json` retains drafts and `pendingReviewId`; the PR view banner explains the closed-PR situation; "Discard all drafts" is offered. On PR re-open, the user is prompted before the existing pending review is adopted (per `spec/03-poc-features.md` § 5 "Drafts on a closed or merged PR" + § 6 idempotency).

### Architectural
- [ ] `using Octokit;` does not appear in any source file under `PRism.Core` or `PRism.Web`. This is a **source-level** dependency-hygiene rule for testability, not a binary-level rule — Octokit is referenced transitively through DI registration in `PRism.Web/Program.cs` (`services.AddSingleton<IReviewService, GitHubReviewService>()` requires the `PRism.GitHub` assembly to be linked, and that assembly carries Octokit) and ships in `PRism.Web`'s output directory at publish time. The DoD does not promise a binary-level isolation; it promises that no `PRism.Core` or `PRism.Web` source file talks to GitHub via Octokit's API surface directly. (Not a multi-provider abstraction promise either.)
- [ ] Every AI seam interface exists with a no-op default implementation.
- [ ] `/api/capabilities` returns `false` for every `ai.*` capability.
- [ ] Every frontend AI slot exists in the component tree but renders `null` when its capability flag is `false`.

### Cross-platform
- [ ] Token storage uses MSAL Extensions (DPAPI on Windows, Keychain on macOS).
- [ ] All file paths use `Environment.SpecialFolder` resolution; no hardcoded `%APPDATA%` or `~/...`.
- [ ] Self-contained single-file binaries publish cleanly for `win-x64` and `osx-arm64`. **`osx-x64` (Intel Mac) is explicitly out of scope for the PoC** — the target audience's Intel-Mac population is small, and shipping a published-but-untested binary is the kind of detail that produces a 100% failure rate when discovered. v2 may revisit if Intel-Mac demand materializes; until then the PoC's distributed binaries are exactly two.
- [ ] First-run setup screen on macOS explicitly tells the user to expect a Keychain "Always Allow" prompt for the unsigned binary (and that it disappears once code signing ships in P4-K4). Verified by manually running an unsigned build on macOS and reading the setup copy.
- [ ] First-run setup screen on Windows explicitly tells the user to expect a SmartScreen "unrecognized app" warning for the unsigned binary, with steps to proceed (More info → Run anyway). Verified by manually running an unsigned build on Windows from a freshly-downloaded location.

### Tests (automated)
- [ ] **Submit pipeline** has tests for: (a) empty-pipeline finalize (verdict + summary, no threads/replies); (b) retry from each step with simulated mid-step failure; (c) foreign-pending-review prompt — Resume path imports server threads as drafts; (d) foreign-pending-review prompt — Discard path deletes server review and starts fresh; (e) commitOID-mismatch on retry triggers discard-and-recreate; (f) foreign-author thread deletion mid-retry demotes the reply to stale.
- [ ] **Reconciliation algorithm** has tests for each of the **nine classification branches** in `03-poc-features.md` § 5: Fresh; Fresh-but-ambiguous; Moved (single match); Moved-ambiguous; whitespace-equivalent → Fresh; whitespace-equivalent multi-match → Moved-ambiguous; no match → Stale; **force-push history rewrite, anchored SHA unreachable, exactly one match in new file → Moved**; **force-push history rewrite, anchored SHA unreachable, multiple matches → Stale**. Plus the file-resolution edge cases: file renamed (via `renamed` status); file deleted. (The two force-push outcomes are separate test cases because a single match re-anchors with a "verify" badge while multiple matches block submit; the failure modes diverge and need distinct coverage.)
- [ ] **State migration framework** has tests for: a successful `vN → vN+1` migration; a *failing* migration leaves `state.json.v{n}.bak` intact and refuses to load.

### Quality
- [ ] Keyboard shortcuts (j, k, v, n, p, c, Esc, Cmd/Ctrl+Enter, Cmd/Ctrl+R, ?, Cmd/Ctrl+/) all work. Cheatsheet opens via `?` outside text inputs and `Cmd/Ctrl+/` from anywhere (including inside composers); closing it preserves composer content and focus.
- [ ] App theme honors `config.json`'s `ui.theme` field with values `light`, `dark`, `system`. Switching the value via config edit (with hot-reload) updates the rendered theme without restart, including code highlighting (Shiki) and Mermaid diagrams. Verified by manually toggling each value and observing the change.
- [ ] No layout shift when a PR with new commits arrives (banner appears without pushing content). **Verification**: a Playwright/Cypress test captures viewport screenshots before and after a simulated banner trigger; bytes-equal except in the banner's reserved sticky region. Run as part of CI for any PR that touches the PR detail view.
- [ ] Accessibility baseline: semantic HTML for headers/landmarks, keyboard-navigable file tree, ARIA labels on all icon-only buttons.

## What "shipped" means for v2

Unlike PoC, v2 has no monolithic Definition of Done — it is an evolving roadmap whose tiers (`backlog/01-P0-foundations.md` through `05-P4-polish.md`) ship incrementally. Three incremental milestones are useful to name:

- **v2-foundations done** = P0-1, P0-2, P0-5, P0-7 shipped (LLM provider, cache, prompt-injection defenses, MCP server). At this point AI features can be built but none has shipped.
- **v2-first-light done** = at least one P1 feature shipped end-to-end (P1-1 PR summarizer is the recommended first because it is highest-signal-to-effort). At this point users see AI augmentation in the PR view.
- **v2-headline done** = P2-2 (PR chat with repo access) shipped, plus one each of the P2 write-side features (composer assist, validators, draft suggestions, draft reconciliation). At this point the wedge thesis "human-led review with AI on the side" is fully demonstrable.

There is no "v2 is finished" — items past v2-headline are continuous polish + new P3/P4 features. A team approaching v2 picks one of these milestones as the immediate goal and uses the backlog's dependency graph to schedule the work.

## What "shipped" means for the PoC

A self-contained binary per platform (Windows x64, macOS Apple Silicon), distributed however the user prefers (direct download, git source for technical colleagues). No installer, no auto-update, no signing/notarization. First user installs Claude Code separately if and when v2 AI features are released — PoC has no such prerequisite. (macOS Intel / `osx-x64` was an earlier publish target; it was dropped because PoC has no test path for it and shipping an untested binary is worse than not shipping one. Apple Silicon is the only macOS target the PoC promises.)

## A note on PoC scope vs. validation cost

The PoC scope — full GitHub integration + GraphQL pending-review pipeline + 7-row stale-draft reconciliation + the AI seam scaffold (16+ Noop classes, ~25 placeholder DTOs, 9 frontend slots) + cross-platform self-contained binary + state migration framework + forensic event log + automated tests for submit/reconciliation/migration — is a **5–8 month commitment for one full-time senior developer** (per `04-ai-seam-architecture.md` § "A note on PoC effort"). The author-dogfood + N=3 external validation gate is run *after* this work; clearing the gate justifies further investment.

**The PoC commits to the full scope.** A leaner alternative was considered (defer AI seams + migration framework + forensic log; ship in 6–8 weeks) and rejected: the seams pay back in v2 (no Core refactor when AI features ship), the reconciliation matrix is hard to retrofit later, and the polish items (migration framework, forensic log) compound the longer they sit. The trade is that ~6 months is at risk against a wedge thesis that hasn't been pressure-tested by a non-author. That risk is acknowledged here (and in "Known risks to the wedge" below) and accepted.

## What success looks like 90 days after the PoC ships

Two signals, weighted differently:

**Author dogfood (necessary, not sufficient).**
The author of the tool prefers using it over GitHub's web UI for reviews. They've used it on 10+ real PRs. They've identified concretely which v2 features (from the backlog) would deliver the most value next, based on real usage rather than speculation.

**The AI workstream (P0–P2) starts only when *both* the author-dogfood signal *and* the external-validation signal pass.** Author dogfood alone is necessary but not sufficient — the author is the most motivated-reasoning-prone observer in the system, and their preference for the tool they built isn't independent evidence. The external N=3 signal (defined below) is the harder, more pressure-checking test; it must clear before AI work justifies its cost. **P3 (multi-platform adapters) and P4 (quality-of-life) work also gate on both signals.** Earlier wording allowed a "some other validating evidence emerges" valve for non-AI work; that valve is removed because it was self-administered by the same author the gate exists to defend against. The discipline is uniform: if both signals haven't passed, no v2 workstream starts. If the project wants to keep building anyway, it has to be honest with itself about the absence of validation rather than declaring an ad-hoc category that bypasses the gate.

**External validation (the harder test).**
**At least three engineers other than the author**, drawn from diverse review habits (e.g., one heavy CodeFlow user, one mostly-github.com user, one IDE-PR-extension user) try the tool on real reviews for at least 5 PRs each. **Audience scope: single-account-or-single-host reviewers only.** PoC supports one `github.host` per launch (cloud OR a single GHES instance), and multi-account / multi-host is a P4 backlog item (`02-architecture.md` § "One host per launch"). The wedge audience is heavily represented in the multi-account population (personal github.com + work GHES, contractor scenarios, OSS maintainer + employer split), and recommending "run two instances" is friction-equivalent to "use two different tools" — those reviewers will default to "I went back to GitHub.com" not because the wedge fails but because the tool's scope rules them out. Run the gate against reviewers whose primary review activity fits a single host so the experiment isn't running against an unfair audience. The multi-account gap is acknowledged separately in `Known risks` (see § "Multi-account / multi-host friction" below).

The success bar: after each trial, the engineer says one of:
- "I'd switch to this if it were available." (Strong signal — counts.)
- "I'd use it for the iteration tabs / file-by-file view but I'd want X before switching." (Useful signal — X becomes a near-term backlog item; counts toward "would use".)
- "I went back to GitHub.com." (Wedge thesis is weaker than the author thought.)

Responses that don't map cleanly to one of three (e.g., "I'd use it for medium PRs, not large ones," "I'd use it once a month not daily," "I'd use it if my team adopted it"): **ambiguous answers count as 'I went back to GitHub.com'**. The conservative posture is deliberate — the gate exists to defend against motivated reasoning by the author who categorizes responses, and "ambiguous → would use" is the easiest place for motivated reasoning to creep back in. If a real reviewer's actual answer is "yes with conditions," they will make that explicit in language closer to the second bullet; the absence of a clear "yes" should be read as a "no" for the purpose of clearing this gate.

Author dogfood is vulnerable to motivated reasoning — the author *built it*, of course they'll like it. A single external user is too easy a bar to clear (one polite friend confirms nothing). N=3 across diverse habits is the minimum sample size where the signal can survive one outlier.

**Gate passes if: at least 1 of 3 lands as "I'd switch to this" (strong yes) AND at least 2 of 3 do not land as "I went back to GitHub.com" (no).**

This is softer than an earlier "≥2 of 3 strong-or-qualified yes" formulation that an adversarial review flagged as structurally near-unreachable (a 67% strong-or-qualified-positive conversion rate is rare for excellent products, especially when one of the three is a github.com loyalist whose priors favor github.com). The softer gate is still meaningful: a single strong yes is hard to fake; two of three avoiding "back to GitHub" is real signal; together they require both **novelty** (somebody would switch) and **broad acceptability** (most aren't actively rejecting).

The external trial does not need to be polished or anonymous; colleagues the author can sit next to are fine. What matters is that the tool faces users who didn't write the spec and didn't form the opinions about what reviewers want.

## Known risks to the wedge

These are recorded so future maintainers can pressure-test against reality, not surprise themselves with them.

- **Claude Code CLI is a moving target.** The v2 AI architecture rests on `claude -p`, `--input-format stream-json`, `--mcp-config`, `--allowedTools` behaving as documented and remaining accessible via subprocess. Anthropic ships frequent CLI updates with no formal API-versioning. The dependency is more fragile than "we shell out to a CLI" suggests. Mitigation: P0-1's empirical-test gate (see `backlog/01-P0-foundations.md` § P0-1) establishes a baseline; any future feature that depends on undocumented CLI behavior must add its own gate.
- **GitHub may copy the wedge.** First-class iteration tabs, file-by-file diff with proper navigation, stale-draft reconciliation — these are features GitHub has shipped variations of in CodeFlow / VS Code's PR extension and could ship in the main UI in 2026. If they do, the wedge that's most heavily leaned on (iteration tabs) evaporates. Mitigation: the *combination* of wedges is harder to copy than any single one (iterations + reconciliation + atomic submit + AI seams), and the local-first posture is structurally distinct from a hosted product. The spec accepts that the wedge is a moving target and depends on shipping AI features faster than GitHub commoditizes the structural ones.
- **The "human reviews AI-authored code" trajectory may shorten.** See "A note on the trajectory" above. If the dominant pattern becomes "AI reviews, humans approve aggregations," the primary surface here is built for a shorter window than the spec assumes. Mitigation: the AI seams do not lock the tool into "human is primary"; a v2 `IPreSubmitValidator` can render an AI-reviewer surface where the human approves rather than authors comments. **Honest acknowledgment: the mitigation is gestural.** The entire UI (file-by-file diff, j/k navigation, inline composer, banner-not-mutation) is shaped for "human reads diff, AI assists at the margins." A pre-submit validator running on drafts the human already wrote is not "AI reviews, human approves" — it's "human reviews, AI sanity-checks." If the trajectory dominates, the surface is wrong and the tool would need a different UI altogether. The cost of this bet is the months building the human-led UI; the seam scaffolding is recoverable, the UI is not. **This is the spec's most consequential strategic uncertainty** — and it is the strongest argument for the lean-PoC option above (validate the human-led-review premise externally on a stripped-down build before investing 6 months in the full surface).
- **GitHub-coupled architecture is intentional.** PoC and v2 commit to GitHub as the only backend (cloud + GHES). There is no provider abstraction, no Azure DevOps / GitLab / Bitbucket adapters in the roadmap. The risk that came with the earlier multi-provider abstraction (paying tax for an abstraction never validated by a second adapter) is gone; the new risk is the inverse — if demand for a non-GitHub backend ever appears, retrofitting it is a meaningful refactor (~6–8 weeks of *shaped* work, not "implement the interface"). That trade is accepted: the target audience is GitHub shops; ADO/GitLab support has zero validated demand from the maintainer's circle; building for it now would be speculative.
- **Multi-account / multi-host friction.** The wedge audience — engineers heavy enough on review to leave github.com — is overrepresented in the multi-account population (personal github.com + work GHES, contractor with several GHES customers, OSS maintainer + employer). PoC's "one host per launch" rule (per `02-architecture.md` § "One host per launch") asks these users to run two PRism instances against two `<dataDir>` values to cover both hosts — practical friction equivalent to "use two different tools." This bites the wedge audience disproportionately, but multi-host is a substantial design problem (a single inbox aggregating PRs across hosts requires per-host auth state, per-host polling budgets, per-host rate-limit accounting, and a UI for distinguishing per-host PR rows). The current trade is to ship single-host PoC, validate the wedge against a single-host audience (per the validation-gate scoping above), and treat multi-host as a backlog item that lights up if the validation gate clears. The risk is that multi-account users self-select away from the validation gate and the gate's signal is therefore narrower than the audience the production tool would target.
- **Claude Code as the AI substrate is single-vendor lock-in dressed as abstraction.** `StreamingSessionOptions` (in `04-ai-seam-architecture.md`) carries `AddDirs`, `AllowedTools`, `DisallowedTools`, `ResumeSessionId`, `McpConfigPath` — these are Claude-Code-specific concepts. An `AnthropicApiLlmProvider` or `OllamaLlmProvider` would either ignore most of them (interface lies about the contract) or implement awkward shims (`AddDirs` for an HTTP API has no meaning). The "Why MCP, not a CLI synthetic-tool flag" rationale is honest about the architectural pin to Claude Code (`--mcp-config` is the only documented path). Mitigation is reactive (CI compatibility test suite). If Anthropic breaks compatibility or moves Claude Code to hosted-only, the seam does not actually decouple from it — the seam is *shaped* for the substrate.
- **"Local-first" is partially cosmetic once AI ships.** Principle 5 says local-first means "the tool never sends data anywhere the user hasn't explicitly authorized for a specific feature." Honest reading: with Claude Code as the only practical AI substrate, "AI features enabled" means "PR diffs, titles, descriptions, and comments are sent to Anthropic." Users in audiences where this is unacceptable (regulated industries, code under NDA, organizations with no Anthropic agreement) cannot use AI features without violating policy. For them, the v2 workstream produces nothing they can use; what they get is the PoC's surface.

  **The Ollama path stays at P4 by deliberate choice, not by oversight.** Ollama (P4-N4) preserves local-first in spirit and is the only architectural answer for the audiences above. It is *not* promoted to P2/P3 because the headline AI feature (chat with repo access via MCP, P2-2) is structurally Claude-Code-specific: the design depends on `--mcp-config`, `--allowedTools`, and the stream-json protocol, none of which Ollama speaks. Re-architecting chat to work over both substrates would roughly double the AI-feature engineering cost across P2 and force every per-feature design (composer assistant, validators, draft suggester) to degrade gracefully on Ollama from day one. The trade we're explicitly taking: PoC users get pure local-first; v2 *non-AI* users (the majority for the first 12+ months) keep local-first; v2 AI users opt in to a third-party data flow at the moment they enable any `ai.*` feature; users for whom that opt-in is unacceptable get the PoC's full surface and no AI. The "local-first" framing is doing more marketing work than substantive work for the v2 AI-enabled-user case — the tool's *config* is local, but the tool's *AI features* depend on a third party. We accept this and document it here so the next maintainer is not surprised by it.
