## 1. Problem & context

P0 (PR1‚ÄìPR3a) built the AI substrate dark: `ILlmProvider` + `ClaudeCodeLlmProvider` (one-shot `claude -p` with the cache levers), the tri-state capability model (`AiCapabilityResolver`, `AiSeamSelector`), `PromptSanitizer`, `ITokenUsageTracker`, `GetDiffAsync`, the event bus, and the `IPrSummarizer` seam with Noop/Placeholder impls. PR3a shipped the **Off | Preview** selector and the SampleBadge sample-data treatment. Every `ai.*` capability still resolves false: no live-capable seam is registered, and the selector's Live branch is **dead** (`liveAvailable: () => false`, hardcoded in `AddPrismAi`).

**This slice lights the first real feature.** It registers a real `IPrSummarizer` (`ClaudeCodeSummarizer`) into the live-seam dictionary, makes the selector's Live branch *live* (the `() => false` becomes a real availability-and-consent check ‚Äî itself a behavioral change beyond "register a seam"), and rewires the PR-detail summary card to a live fetch. It is the roadmap's deliberate **lowest-blast-radius** first surface: read-only, free-text, one PR at a time, user-verifiable against the diff.

**Live enablement is folded in as the UI prerequisite, in the correct order.** A live summary call egresses PR content (diff, title, description) to Anthropic via the Claude Code CLI. PR3a's spec flagged this as a hard sequencing dependency: *"PR3b's egress-consent gate MUST land before any P1 work registers a real Live seam ‚Ä¶ a hand-edited Live config could egress PR content with no consent."* This slice resolves that risk **within the same change**.

**Atomic-ordering mandate (load-bearing for safety).** The consent term MUST be wired into the gating predicates (¬ß5) **in the same PR** that registers `ClaudeCodeSummarizer`. The implementation plan must not split these across PRs: a window where the real seam is registered but the consent predicate is absent would let a direct `POST /api/preferences {"ui.ai.mode":"live"}` egress PR content with no consent. **Structural enforcement (so the mandate is not prose-only):** the `liveAvailable`-delegate flip and the `ClaudeCodeSummarizer` registration are the **same edit site** in `AddPrismAi` ‚Äî they cannot be split without an obviously incomplete diff ‚Äî and the exit test "Live + no consent ‚áí 204, zero egress" MUST be authored in that same PR. That exit test (¬ß12) is a **required**, blocking criterion ‚Äî not advisory.

## 2. Scope & non-goals

**In scope:**

- **Backend** ‚Äî `ClaudeCodeSummarizer : IPrSummarizer` (provider + sanitizer + diff + token tracker + a per-process in-memory summary cache); register it into the live-seam dictionary; the explicit D111 `IsSubscribed` gate on `/ai/summary`. The summarizer also emits a **minimal PR-nature category** on the same call (a free rider ‚Äî no extra request, no extra tokens ‚Äî **accept-drift**, ¬ß4, ¬ß13).
- **Backend** ‚Äî egress consent wired into the Live-availability gating (the two predicates + the disabled-reason path ‚Äî see ¬ß5 for the *actual* wiring, which is more than a one-line change); a consent record in `AppConfig`; `GET /api/ai/egress-disclosure`; `POST /api/ai/consent`.
- **Backend** ‚Äî the **per-feature user-enablement seam** (¬ß5.1): a per-feature `userEnabled` term threaded into the gating so a user can switch an individual AI feature off and stop its token spend, backend-enforced. Storage + holder + the gate term land now with **Summary** as the first consumer (default on); the per-feature **toggle UI** and the FE per-flag wire are deferred (below). This exists from the get-go so P2's inbox enrich/rank (the real token spenders) inherit the gate with zero backend retrofit.
- **Frontend** ‚Äî make **Live** a selectable third segment via a two-phase commit (¬ß7); the consent-before-flip `EgressConsentModal`; rewire `AiSummaryCard`/`useAiSummary` with a loading affordance + a no-retry error state (¬ß13 resolved), coordinated with the active-PR subscription.

**Out of scope ‚Äî deferred:**

- **P1b** ‚Äî `<dataDir>/llm-cache` file cache + restart survival, event-bus eviction, the per-PR **context artifact** keyed `(prRef, baseSha, headSha)`, **base-rebase staleness** eviction (R2), measured prompt-cache hit, the **stale badge + Regenerate** UX, the formal **`IAiCache`** contract (introduced when the disk impl's requirements are known ‚Äî ¬ß4), and the disclosure-version **409 re-fetch** path (¬ß7).
- **The inbox classifier surface + cross-surface reconciliation** ‚Äî P1a *does* ship a minimal **detail-card** category (free rider on the summarizer call, accept-drift ‚Äî ¬ß4, ¬ß13). What's deferred is the **inbox** half of "one feature, two surfaces" (the metadata-batch classifier feeding the inbox chip, roadmap ¬ß3.2) **and** the reconciliation that would make the two surfaces agree (a single authoritative category cached on the per-PR context artifact, P1b/P2). Until then the two surfaces compute independently and may disagree on the same PR.
- **The 4 disabled-state guidance messages**, the **per-feature toggle UI**, and the **FE per-flag wire (D112)** all land with a **dedicated Settings ‚Üí AI pane** ‚Äî AI mode stays in `AppearancePane` this slice; consent is a modal triggered from there. Because every feature defaults `userEnabled=true` (¬ß5.1), the FE needs no per-flag knowledge yet (all-on), so D112 stays deferred without retrofit risk.
- **The re-consent affordance** for the "Live + consent-required" state ‚Äî unreachable through normal UI this slice (single provider, constant disclosure version); recovery is the Off‚ÜíLive re-toggle (which re-opens the modal). Built when a real trigger ships.
- **Multi-provider** ‚Äî there is one provider. The "active provider id" is a constant (¬ß5); per-provider consent switching is modeled in the record shape but is **forward-looking** (no switch mechanism exists to exercise it).

> **Scope decisions resolved (¬ß13):** both the failure-UX (no-retry inline error) and the minimal-category (accept-drift) questions are resolved below. No open scope decisions remain before planning.

## 3. Architecture overview

```
PR detail (Live) ‚îÄ‚îÄGET /ai/summary‚îÄ‚îÄ‚ñ∫ AiEndpoints (handler gains IActivePrCache)
                                         ‚îÇ  D111: IsSubscribed(prRef)?  no ‚îÄ‚ñ∫ 204
                                         ‚ñº
                                    IAiSeamSelector.Resolve<IPrSummarizer>()
                                         ‚îÇ  Live branch: seamRegistered
                                         ‚îÇ            && liveAvailable()   ‚Üê was () => false; now real
                                         ‚îÇ            && consentRecorded(providerId)
                                         ‚îÇ            && userEnabled(feature)   ‚Üê per-feature toggle (¬ß5.1)
                                         ‚îÇ  not usable ‚îÄ‚ñ∫ NoopPrSummarizer ‚îÄ‚ñ∫ null ‚îÄ‚ñ∫ 204
                                         ‚ñº
                                    ClaudeCodeSummarizer.SummarizeAsync(prRef)
                                         ‚îÇ  in-memory cache (prRef, headSha) hit ‚îÄ‚ñ∫ cached (0 provider calls)
                                         ‚ñº  miss
              IPrReader.GetDiffAsync(prRef, DiffRangeRequest) ‚îÄ‚ñ∫ sanitize diff + title + description
                                         ‚îÄ‚ñ∫ ILlmProvider.CompleteAsync (v1 prompt, --append-system-prompt)
                                         ‚îÄ‚ñ∫ ITokenUsageTracker.RecordAsync(feature:"pr-summary")
                                         ‚îÄ‚ñ∫ PrSummary(Body, Category="")  ‚îÄ‚ñ∫ 200
```

**Consent is wired into the gating, but it is NOT "one shared predicate" today ‚Äî be precise.** The current code has *two structurally different* Live-availability inputs plus a *third* reason path:

1. `AiSeamSelector.Resolve<T>()` gates Live on a `Func<bool> _liveAvailable` (today `() => false`). Consent must be folded into this delegate, and the delegate must start returning real availability.
2. `AiCapabilityResolver.Capable(seam)` gates Live on a per-call `LlmAvailability.Available`. Consent must be incorporated here too.
3. `AiCapabilityResolver.DisabledReason(mode, availability)` is a **static** method whose signature has no consent input, and `CapabilitiesEndpoints` injects no consent source. Emitting `disabledReason="consent-required"` requires (a) extending `DisabledReason` to take consent state (or making it instance state reading `AiConsentState`) and (b) injecting the consent source into `CapabilitiesEndpoints`.

So the consent change touches **three sites**, not one. The reason-precedence rule must be explicit: when Live is **both** probe-unavailable **and** unconsented, `disabledReason` reports the **provider/probe reason first** (consent is moot if the provider can't run); `consent-required` is reported only when the provider is available but consent is absent.

**The full effective gate** is `effective(feature, mode) = mode‚âÝoff && userEnabled(feature) && capableForMode(feature, mode)`, where `capableForMode` in Preview = a Placeholder exists (always true) and in Live = `seamRegistered && available && consented`. `userEnabled(feature)` (¬ß5.1) is the new per-feature term and gates **both** Preview and Live presence (turning a feature off hides its sample too); Off is all-off regardless. For P1a every feature defaults `userEnabled=true`, so Summary's behavior is unchanged ‚Äî the term is the seam, exercised by one feature now.

## 4. Backend ‚Äî the summarizer

**`ClaudeCodeSummarizer : IPrSummarizer`** (new). **Composition site:** compose it in `PRism.Web`'s `AddPrismAi` (where the AI seams are already wired), not inside `PRism.AI.ClaudeCode` ‚Äî that keeps the AI provider project free of a `PRism.Core.PrDetail` dependency. `SummarizeAsync(PrReference, ct)`:

1. **Resolve the diff + headSha.** Derive the diff and the current `headSha` from the **diff-fetch / `PrDetailLoader` path** (`PrDetailLoader.TryGetCachedSnapshot` / `GetOrFetchDiffAsync`, reusing the memoized diff). **Do not source SHAs from `ActivePrSnapshot`** ‚Äî it carries `HeadSha` only (no `baseSha`) and is null until the first poll. The cache key is `(prRef, headSha)`. **`baseSha` IS available** on this path ‚Äî `snapshot.Detail.Pr.BaseSha`, the producer `PrDetailEndpoints` already uses to build a `DiffRangeRequest` (PrDetailEndpoints.cs:85). Keying on `(prRef, headSha)` only is therefore a **deliberate P1a scope cut** ‚Äî it avoids base-change eviction machinery this slice ‚Äî **not** a "no producer exists" limitation; `(prRef, baseSha, headSha)` keying is achievable from the snapshot and is the R2/P1b target (¬ß9). Use the live `GetDiffAsync(prRef, DiffRangeRequest)` overload ‚Äî the legacy `(fromSha, toSha)` overload throws `NotImplementedException`.
2. **Cache lookup.** A per-process in-memory cache keyed `(prRef, headSha)`. Hit ‚áí return cached `PrSummary`, **zero provider calls**. *For P1a this may be the BCL `IMemoryCache` or a small internal dictionary; the formal `IAiCache` contract is introduced in P1b when the disk impl's contract needs (restart, eviction hooks) are known ‚Äî introducing a one-impl interface now is premature.*
3. **Sanitize all attacker-controlled inputs.** `PromptSanitizer.WrapAsData(diff, "diff")`, `WrapAsData(title, "title")`, `WrapAsData(description, "description")`. **PR title and description are author-controlled and MUST be wrapped** ‚Äî not just the diff (the sanitizer's own contract lists titles as attacker-controllable).
4. **Generate.** `ILlmProvider.CompleteAsync` with a hand-written **v1 system prompt** via `--append-system-prompt` (cache-prefix-stable). The single call returns **both** the body and a minimal category ‚Äî a free rider, no extra request or tokens. **Body** = free-text summary. **`Category`** = one value of a fixed taxonomy `{feature, fix, refactor, docs, test, chore, revert}` (committed default; product-owner-overridable), parsed from a leading `CATEGORY: <value>` line the prompt instructs the model to emit; the parser validates the value **case-insensitively against the enum** and **strips the line from the body**. A missing / out-of-enum / low-confidence value ‚áí **`Category = ""`** (the no-confident-category fallback ‚Äî rendered body-only, ¬ß7). Keying is unchanged: the category rides inside the cached `PrSummary` on `(prRef, headSha)`, so a head change re-summarizes and re-categorizes together.
5. **Record usage.** `ITokenUsageTracker.RecordAsync(new TokenUsageRecord(Feature:"pr-summary", ‚Ä¶))`.
6. **Cache store** keyed `(prRef, headSha)`, then return. **Failures are not cached:** a provider exception propagates from step 4 *before* this store is reached, so the cache holds successes only. This is what makes the ¬ß7/¬ß13 "reopen the PR to recover" path real ‚Äî reopening re-resolves the same `(prRef, headSha)` and re-invokes the provider rather than serving a cached failure. Covered by a unit test (a post-failure resolve re-calls the provider).

**Minimal category ‚Äî accept-drift (resolves the ¬ß13 open decision).** The detail-card category costs nothing extra (free rider on the call) and lets the first AI card read as intentionally categorized rather than body-only. It is computed from the **full diff**, whereas P2's inbox category (roadmap ¬ß3.2) is computed from **metadata-only batches** ‚Äî so the two surfaces can disagree on the same PR. That drift is **accepted in P1a**: the surfaces compute independently and are not reconciled. Reconciliation (a single authoritative category cached on the per-PR context artifact and reused by the inbox) is deferred to P1b/P2 ‚Äî pulling it forward would drag the P1b context artifact into P1a and dissolve the P1a/P1b split. The taxonomy is pinned **now** because both surfaces should show the same buckets; if the inbox later needs different buckets, that is a taxonomy migration ‚Äî the accepted forward risk of folding category in early.

**Provider id.** Consent (¬ß5) keys on a provider id. There is one provider; use a backend constant `ClaudeProviderId = "claude-code"` (matching the literal already used in `TokenUsageRecord`). The multi-provider registry is deferred.

**Registration.** Add `ClaudeCodeSummarizer` to the live-seam dictionary **and** flip the selector's `liveAvailable` delegate to a real availability+consent check, in the same PR (¬ß1 mandate).

**Eval.** P1a ships a reasonable v1 prompt. The golden-set tuning loop (roadmap ¬ß11.1) is the **P1‚ÜíP2 gate**, human-anchored ‚Äî it does **not** block this slice's merge.

## 5. Backend ‚Äî egress consent & per-feature enablement

**Predicates (the real wiring ‚Äî see ¬ß3).** Consent term: `consentRecorded(ClaudeProviderId)` ‚áî a stored record exists whose `DisclosureVersion == current` and whose `ProviderId == ClaudeProviderId`. Fold it into (1) the selector's `liveAvailable` delegate and (2) the resolver's `Capable`, and surface it via (3) `DisabledReason` + `CapabilitiesEndpoints`. No consent ‚áí `Summary` flag `false`, `disabledReason="consent-required"` (when provider available), seam ‚áí Noop ‚áí 204. **Security guarantee:** a direct `POST /api/preferences {"ui.ai.mode":"live"}` cannot egress ‚Äî the seam resolves Noop until a consent record exists.

**Storage.** Add an `AiConsentConfig` record (`{ ProviderId, DisclosureVersion, AcknowledgedAt }`) to `AppConfig`. **Placement + backfill (avoids a startup NRE):** add it as an explicit member and add the matching null-backfill in `ConfigStore.ReadFromDiskAsync` (`AiConsent = parsed.AiConsent ?? AppConfig.Default.AiConsent`), mirroring the existing `Ui`/`Llm`/`Polling` backfills. `Default.AiConsent` represents "no consent recorded" **concretely** ‚Äî a record with `DisclosureVersion = null` (and `ProviderId`/`AcknowledgedAt` null), so the `consentRecorded` predicate (`DisclosureVersion == current && ProviderId == ClaudeProviderId`) evaluates **false** on it. Persist via a **dedicated `ConfigStore.RecordAiConsentAsync(providerId, disclosureVersion, ct)`** (modeled on `SetDefaultAccountLoginAsync`, *not* the flat `PatchAsync` allowlist) ‚Äî and cover it with a concurrent-write test (a consent write racing another config mutation must not be lost). Mirror into an `AiConsentState` singleton on the `ConfigStore.Changed` event, as `ui.ai.mode` mirrors into `AiModeState`.

**Disclosure source.** `RecipientIdentity = "Anthropic, via the Claude Code CLI"` and `DataLeavesDevice = true` are constants owned by the **egress-disclosure module / endpoint** (one consumer), **not** added to `ProviderCapabilityDescriptor` ‚Äî that record documents itself as "minimal until a second provider lands," and two always-true-for-the-only-provider fields would violate that invariant.

**Disclosure version.** Backend constant `DisclosureVersion = "1"`; consent valid only while `record.DisclosureVersion == current`. **A "material change" that warrants a bump** = a change to the recipient, the data categories sent, or retention/usage terms (not copy-editing). Bumping invalidates stored consent ‚áí re-prompt. Switching the active provider (forward-looking) invalidates likewise. Both fall out of the predicate.

**Change-control rule (so a missed bump can't silently widen egress).** The version field is the entire integrity of the consent model: a future change that sends *more* data (the spec already defers adding `baseSha`, comment context, etc. to P1b+) without a matching bump would silently authorize the expanded egress under a stale `v1` consent. Enforcement: the `dataCategories` list (below) and the actual fields the summarizer sends are the single source of truth; a PR that changes **what leaves the device** MUST bump `DisclosureVersion`. Add this as a PR-template / review-checklist item ("did you change the data sent to the provider? ‚Üí bump `DisclosureVersion`"). Engineering review is the catch; no separate owner is assigned in P1a.

**Endpoints (both new; both under the existing `/api/*` pipeline ‚Äî `SessionTokenMiddleware` requires a session token; `POST` is additionally covered by `OriginCheckMiddleware`):**

- `GET /api/ai/egress-disclosure` ‚Üí `{ recipient, dataCategories[], disclosureVersion, alreadyConsented }`. `dataCategories` = `["Pull request diff (changed files and their contents)", "Title", "Description"]` ‚Äî truthful to what the summarizer sends. `alreadyConsented` = active provider has a valid current-version record (lets the FE skip the modal on repeat Live selection).
- `POST /api/ai/consent { disclosureVersion }` ‚Üí if `disclosureVersion != current` ‚áí **409**. Else stamp `{ ClaudeProviderId, disclosureVersion, AcknowledgedAt = UtcNow }`, return **204**. The backend stamps the provider id itself. Re-recording is idempotent (re-stamps `AcknowledgedAt`).

**Consent withdrawal (named gap).** Switching to Off/Preview does *not* delete the consent record ‚Äî it makes the predicate moot until Live is re-selected. A permanent "revoke consent" action is **not** in this slice (deferred with the Settings‚ÜíAI pane); document this so it is a conscious omission.

### 5.1 Per-feature user enablement (token-spend control)

A second gating axis, distinct from *system capability*: the user's explicit choice to run a given AI feature at all. Motivation is token spend ‚Äî a user who finds (say) inbox enrichment unhelpful must be able to switch it off and stop its calls, and that switch must be **backend-enforced** (a frontend-only toggle would still burn tokens, and the inbox fan-out features are the real spenders). This slice lands the **seam**, not the UI.

- **Storage.** Add an `ai.features` block to `AppConfig` ‚Äî a per-feature `bool` keyed by the nine `AiCapabilities` field names (`summary`, `fileFocus`, `hunkAnnotations`, `preSubmitValidators`, `composerAssist`, `draftSuggestions`, `draftReconciliation`, `inboxEnrichment`, `inboxRanking`). **Default: every feature `true`** (so existing behavior is unchanged). Persisted via `ConfigStore` with the same null-backfill discipline as consent (absent on disk ‚áí all-true Default; no NRE).
- **Holder.** An `AiFeatureState` singleton mirrors `ai.features` on `ConfigStore.Changed`, parallel to `AiModeState` / `AiConsentState`.
- **Gate.** `AiSeamSelector` and `AiCapabilityResolver` consult `AiFeatureState.IsEnabled(featureKey)` (the implementation of the conceptual `userEnabled(feature)` term in ¬ß3's formula ‚Äî same thing) as a term in the effective gate. A disabled feature resolves to **Noop ‚áí 204** in Live (no token spend) and to Noop (no sample) in Preview.
- **Mechanism (the selector signature must change ‚Äî not just a new boolean AND).** Today the selector gates Live on a **parameterless** `Func<bool> _liveAvailable` (`AiSeamSelector.cs:18`, invoked `_liveAvailable()` at line 40) that has no knowledge of which seam `T` is being resolved. A per-*feature* term cannot ride that delegate. This slice changes `AiSeamSelector` to receive `AiFeatureState` (and the consent source) and resolve the feature key from `typeof(T)` via a **seam-type ‚Üí feature-key map** (reusing the capability-flag ‚Üî seam-type correspondence already in `AiCapabilityResolver`) **inside `Resolve<T>()`** ‚Äî a constructor/shape change to the selector, made once here. The same change is what carries the consent term (¬ß5), so both land in the one edit (¬ß1 mandate); the plan must treat the selector ctor change as the integration point, not a late add.
- **This slice wires Summary only** (`userEnabled("summary")`, default true). The other eight keys exist in storage and are gated, but only Summary has a live consumer to exercise the path. No `POST` to set per-feature flags and no FE wire ship here ‚Äî the Settings‚ÜíAI toggle UI owns those (deferred). Because all default true, `/api/capabilities` and the FE need no change this slice.

**Why now:** P2 registers the inbox enrich/rank seams (the high-volume token spenders). If the per-feature gate exists from P1a, those features inherit it by construction. If it does not, P2 would otherwise land the gate later ‚Äî the selector ctor change above **plus** the storage/holder/backfill ‚Äî *as well as* the inbox seams, in one heavier slice. Landing the seam now is a small, deliberate forward-investment (storage + holder + backfill + the one selector change, all default-on so behavior is unchanged), chosen by the product owner to avoid bundling cross-cutting plumbing into the higher-risk P2 inbox work. Recorded as a cross-cutting model in roadmap ¬ß3.2/¬ß4.

## 6. Backend ‚Äî the D111 gate on `/ai/summary`

`IActivePrCache.IsSubscribed(prRef)` is the D111 token-spend gate: a live summary call fires only for a PR with an active subscriber. The `/ai/summary` handler currently injects only `(IAiSeamSelector, ct)`; the gate **adds an `IActivePrCache` parameter** and an early `if (!cache.IsSubscribed(prRef)) return NoContent();` branch **before** resolving the seam, mirroring the established `IsSubscribed` pattern in `PrSubmitEndpoints`/`PrRootCommentEndpoints`. This replaces the standing `D111: ‚Ä¶add an IsSubscribed gate` comment with real code + a test. Off / no-consent / not-subscribed / feature-disabled all converge on **204** (gate closed); only a gate-open provider failure differs (503, ¬ß7).

**Subscription-race coordination (correctness, not just spend).** The active-PR subscription is established by an async SSE handshake (`useActivePrUpdates`: stream connect ‚Üí `subscriberId()` ‚Üí `POST /api/events/subscriptions`). On a cold PR open, a summary fetch fired on mount can **beat** subscription registration ‚áí D111 ‚áí 204 ‚áí card hidden ‚Äî and today `useAiSummary` only refetches on `[prRef, enabled]`, so it never recovers. **Fix:** gate `useAiSummary`'s `enabled` (and refetch) on the active-PR subscription being **established**, not on bare mount, so the FE fetch and the backend D111 gate agree. Tested explicitly (¬ß11) ‚Äî the existing always-subscribed test fake hides this race.

## 7. Frontend

**`AppearancePane` selector ‚Äî two-phase commit (avoids flash-then-revert).** Options become `Off | Preview | Live`. The `SegmentedControl` fires `onChange` on **arrow-key** navigation and moves focus synchronously, so the Live segment must **not** be driven straight off `preferences.ui.aiMode`. `AppearancePane` holds a local **`pendingLive`** state; the control's `value` shows the **resolved** mode (off/preview/live from preferences), never a pending Live. Because the control's role-state follows `value`, the Live segment's `aria-checked` stays **false** throughout the pending window (disclosure GET in flight / modal open), so a screen reader announces Live as not-yet-selected until consent actually commits ‚Äî it never falsely announces Live as active mid-intercept. Flow:

- `off` / `preview` ‚Üí `set('ui.ai.mode', next)` directly (existing behavior + no-op guard).
- `live` ‚Üí **intercept; do not POST, do not advance the control's value.** `GET /api/ai/egress-disclosure`; if `alreadyConsented` ‚áí `set('ui.ai.mode','live')` directly; else open `EgressConsentModal`.
- **Navigate-away while the disclosure GET is in flight** (user arrow-keys/clicks back to Off or Preview before it resolves) ‚Üí **cancel the in-flight request (AbortController) and clear `pendingLive` without opening the modal.** The previously-selected segment is already the control's value, so nothing reverts. A late-arriving response for an abandoned selection must not open the modal.
- **Accept** ‚Üí `POST /api/ai/consent` ‚Üí on 204, `set('ui.ai.mode','live')` (now the control advances) ‚Üí clear pending ‚Üí close ‚Üí **return focus to the now-active Live segment** (the committed action's destination ‚Äî symmetric with Decline returning focus to the prior segment).
- **Decline / error** ‚Üí clear pending, no mode change; the control was never advanced, so "revert" is a no-op and focus returns to the **previously-selected segment** (e.g. Preview), not the Live button that triggered the intercept.
- **Non-204 from consent POST** (incl. 409) ‚Üí treat as generic failure: toast + revert. *(The 409-specific disclosure re-fetch is deferred to P1b with the re-consent affordance ‚Äî building it now serves a trigger that cannot fire this slice.)*

**`EgressConsentModal`** (new; on the existing `Modal` primitive ‚Äî focus-trapped). **Interaction states, all specified:**
- **Title:** "Enable Live AI" (the `Modal` required title / `aria-labelledby`).
- **Value-first copy:** lead with what Live enables (a real, diff-grounded PR summary) **before** the egress disclosure ‚Äî first contact with AI should not be a bare legal wall.
- **Disclosure body:** `recipient`, `dataCategories`, per-call nature, wired via `aria-describedby` (the most important content for screen readers).
- **Loading:** while the disclosure GET is in flight, render the modal with a `Skeleton` body + Accept/Decline disabled + `aria-busy="true"` + a visually-hidden "Loading data-sharing disclosure‚Ä¶" status (the existing `Skeleton` is reduced-motion-safe).

(Copy is a committed default; the product owner may override.)
- **Default focus:** **Decline** (the less-destructive action), matching the discard-modal precedent.

**`useAiSummary` + `AiSummaryCard`.** The hook returns `PrSummary | null` today; refactor to `{ summary, loading, error }` and gate `enabled` on subscription-established (¬ß6). **Consumer + mock migration (both test trees):** update the `OverviewTab` destructure and `AiSummaryCard` prop usage, and every `vi.mock('.../useAiSummary', () => ({ useAiSummary: () => null }))` site ‚Üí `() => ({ summary: null, loading: false, error: false })`, in **both** co-located `src/**` and legacy `frontend/__tests__/`. `AiSummaryCard` renders, by state:
- **loading** ‚áí a `Skeleton`-based affordance (`aria-busy`, matching `InboxSkeleton`) plus a visually-hidden `aria-live="polite"` status reading **"Loading AI summary‚Ä¶"** (mirrors the modal's "Loading data-sharing disclosure‚Ä¶" pattern, ¬ßbelow ‚Äî so a screen-reader user isn't left in silence).
- **success** ‚áí the summary.
- **provider-failure** ‚áí a one-line inline error. **Committed copy:** **"AI summary unavailable ‚Äî reopen this PR to try again."** ‚Äî it names the recovery path so the deliberate, non-spammable recovery is *discoverable* rather than implicit (the ¬ß13 no-retry decision withholds a button, not the information). **No Retry control** and **no auto-retry**; deliberate Regenerate is P1b. (Copy is a committed default; the product owner may override the exact words.)
- **gate-closed (204)** ‚áí hidden.

**`Category` rendering (authoritative ‚Äî supersedes the looser "suppresses the empty label" phrasings in ¬ß2/¬ß4).** A **confident category** (a non-empty enum value) renders as a **category chip** showing the title-cased label ‚Äî reuse the existing inbox category-chip styling/pattern (from PR3a) rather than inventing one. When `Category === ""` (the no-confident-category fallback, ¬ß4), the card renders **body-only: no category-row element and no whitespace reserved for one** ‚Äî the row is absent from the DOM, not a hidden-but-space-holding label, so the card reads as intentionally single-region rather than as a placeholder with a missing field. Both states must look intentional; the empty state is the fallback, not the default-for-the-whole-window.

**Distinguishing failure from absence (required for the error state).** The FE can only show an error if the backend separates "tried and failed" from "not applicable." So: a **gate-closed** outcome (Off / not-subscribed / no-consent / feature-disabled ‚Üí seam is Noop ‚Üí null) returns **204** (hidden); a **gate-open** outcome where the real summarizer's provider call throws returns a **distinct non-success status (e.g. 503, never 500)** that the FE maps to the error state. Mechanically: `NoopPrSummarizer` returns null (‚áí 204); `ClaudeCodeSummarizer` lets `LlmProviderException` propagate (it no longer swallows it to null), and the endpoint catches it ‚Üí 503. Success ‚áí 200.

**Help text.** Update the `ai-mode-help` text to describe all three segments (currently "Off ¬∑ no AI. Preview ¬∑ sample output, clearly labeled." ‚Üí add Live), so the radiogroup's `aria-describedby` stays accurate.

## 9. Error handling & accepted limitations

- **Provider failure/timeout** (gate was open) ‚áí `LlmProviderException` propagates ‚áí endpoint returns **503** ‚áí FE shows the inline **no-retry** error state (¬ß7, ¬ß13). **Never 500.** Gate-closed cases (Off / not-subscribed / no-consent / feature-disabled) return **204** ‚áí card hidden ‚Äî these are not failures.
- **Token-tracker write failure** ‚áí logged, non-fatal.
- **Disclosure fetch failure** ‚áí modal error, cannot consent ‚Äî **fail-closed**.
- **Accepted: base-rebase staleness (R2, roadmap-rated High).** Cache keyed `(prRef, headSha)`; a base rebase leaving `headSha` unchanged can serve a stale summary until process restart. **A GitHub tracking issue for P1b (R2 `(prRef, baseSha, headSha)` keying + base-change eviction, plus the other deferred P1b items) MUST be filed and referenced before P1a merges** ‚Äî so a High risk is not left open without an enforcement trail.
- **Accepted: consent-revocation TOCTOU (one-call window).** Consent is read at seam-resolve time; egress happens ~10s later inside `CompleteAsync`. A revocation in that window lets one in-flight call complete. Blast radius is a single call; accepted under the local-desktop threat model (consistent with the base-rebase acceptance).
- **Accepted: file-crafted consent record bypasses the modal.** The backend predicate guarantees a direct `POST /api/preferences` cannot egress without a record (¬ß10), but a process with write access to `AppConfig` on disk could pre-seed a structurally valid `AiConsentConfig` and skip the modal. Accepted under the **local-desktop threat model**: a process that can write `AppConfig` already owns the machine ‚Äî it could equally flip `ui.ai.mode`, run the app's own keying code, or read the PRs directly ‚Äî so a MAC/signature on the record adds complexity without raising the bar against a local-root attacker. Consent integrity is enforced at the **API boundary**, not against the local filesystem.
- **Accepted: 204-vs-503 status side-channel.** A caller holding a valid session token can infer consent-and-availability state by observing whether `/ai/summary` returns 204 (gate closed) or 503 (gate open, provider failed). Low value under the local-desktop, single-user model (the only such caller is the user's own session); accepted rather than collapsing the error signal back into 204, which would forfeit the distinguishable-failure requirement (¬ß7).
- **Accepted: ~10s latency, no streaming.** Loading affordance covers it. (Observed in T18 smoke: a large 88K-char diff took ~38s end-to-end ‚Äî the claude-code harness overhead on big PRs; a progress affordance / diff-size cap is noted in #379.)
- **AI interaction audit log (added 2026-06-11).** Every AI seam invocation writes one **metadata-only** JSONL line to `<dataDir>/logs/ai-interactions.log` (`IAiInteractionLog` / `JsonlAiInteractionLog`), tagged with the triggering `component` (`"summary"`), plus provider / model / prRef / headSha / outcome (`ok` | `cacheHit` | `providerError`) / `egressed` / latency / tokens / cost / prompt+response **char counts** ‚Äî **never** prompt or response content (those carry the PR diff and possible secrets). Non-fatal: a failed write can never deny a summary already computed (mirrors the token-tracker rule above). This is the egress-accountability complement to the consent gate. **Known gap (issue #379):** `claude-code` reports only the few genuinely-new input tokens (the CLI prompt-caches its own harness) and `LlmResult` has no `CacheCreationInputTokens` field, so the logged token counts under-report by orders of magnitude ‚Äî `estimatedCostUsd` is currently the only reliable spend signal.

## 10. Security

- **Egress consent is backend-enforced** via the predicates (¬ß5), not a cosmetic FE gate. Direct-POST-to-Live cannot egress without a record. The consent predicate and the seam registration ship in **one PR** (¬ß1).
- **All author-controlled inputs sanitized:** diff, title, **and** description via `PromptSanitizer.WrapAsData`.
- **Prompt-injection battery** runs against the **real provider invocation path** ‚Äî i.e. the combined (`--append-system-prompt` keeps Claude Code's default identity in front) prompt as `ClaudeCodeLlmProvider` produces it, not an isolated fragment ‚Äî and includes injection payloads in the **diff, the title, and the description**. The delimiter-escape test is **cross-field on the assembled prompt**, not a single isolated `WrapAsData` call: a malicious diff that tries to terminate its own `WrapAsData` block and forge a `</title>`/`<description>` boundary must not be able to inject content into another field's region. Assert the three blocks' boundaries survive in the fully assembled prompt (all three fields present), since that is where a cross-field escape would land.
- **Forged-category injection is bounded by enum validation.** A payload in the diff/title/description that tries to coerce a wrong `CATEGORY:` line can at worst yield a wrong-but-valid bucket or `""` ‚Äî never arbitrary output ‚Äî because the parser discards anything outside the fixed enum. Tested (injected out-of-enum/garbage `CATEGORY:` ‚áí `Category=""`, body intact).
- **Token-usage records carry no prompt text / secrets** (existing shape).

## 12. Exit criteria

**P1a (this slice):**
- Real summary within ~10s for a ~200-line PR; provider call count **1** on miss, **0** on in-memory cache hit.
- Flag off ‚áí 204; **Live + no consent ‚áí 204 with zero egress** (blocking); not-subscribed ‚áí 204; cold-open subscription race recovers.
- Injected `IGNORE PREVIOUS INSTRUCTIONS‚Ä¶` in diff/title/description does not corrupt output.
- Consent recorded gates Live end-to-end; declining leaves mode unchanged.
- **Per-feature enablement seam present:** Summary gated through `userEnabled("summary")` (default on, behavior unchanged); `userEnabled("summary")=false` ‚áí 204, zero provider calls; the other eight keys exist in storage + gate.
- **Minimal category present:** the summary call returns a validated PR-nature category from the fixed taxonomy on the **same** provider call (no extra call); a confident value renders as a chip, `""` renders body-only; an out-of-enum / forged value resolves to `""` (never arbitrary text).
- **Provider failure surfaces:** gate-open provider failure ‚áí 503 ‚áí inline no-retry error (not a silent hidden card).
- The PR3a sequencing risk is **closed**: consent guard + first real seam in one PR.
- **R2 tracking issue filed and its URL referenced in the PR** before merge (the base-rebase staleness deferral, ¬ß9) ‚Äî a blocking exit item, not advisory, so a roadmap-High risk is not left open without a trail.

**Deferred to P1b ‚Äî and the roadmap ¬ßP1 governance gate depends on them (roadmap: "the dogfood signal gates on P1b"), so merging P1a does NOT clear the P1‚ÜíP2 gate:** `cache_read_input_tokens > 0` (measured prompt-cache hit), `ActivePrUpdated` eviction + regenerate, base-rebase eviction (R2), head-shift-mid-call leaves no stale entry, stale-badge + Regenerate UX. *(Whether `ITokenUsageTracker` should already capture `cache_read_input_tokens` in P1a ‚Äî so P1b can measure without a schema change ‚Äî is a plan-phase question; the field exists on `LlmResult` today.)*
