# PR detail: GitHub lifecycle actions — Slice 1 (foundation + close/reopen/draft-toggle) (#566)

**Status:** spec (T3 — full pipeline; gated **B1 UI-visual** + **B2 risk-surface: auth/PAT + first GitHub write actions**)
**Issue:** [#566](https://github.com/prpande/PRism/issues/566) — PR detail lifecycle actions.
**Sibling:** [#571](https://github.com/prpande/PRism/issues/571) — resolve/unresolve review threads. Shares the write-path foundation built here (owner directive on #566: *"design the UX separately, build the plumbing once"*).

## Problem

The PR detail screen is review-only. You can submit a review and post comments, and the header *displays* draft/open/merged/closed state — but you cannot take **GitHub PR lifecycle actions** (merge, close, reopen, draft↔ready) from inside PRism. Today the only escape hatch is the header's Open-in-GitHub link — you leave for github.com to act. `PRism.GitHub` itself only **reads** PR/merge state (`IPrReader`) and **writes review comments/submissions** (`IReviewSubmitter`); there are no lifecycle write methods.

## Scope: this is Slice 1 of two

Owner-selected slicing (option B): split #566 into two PRs rather than one.

- **Slice 1 (this spec):** the shared write-path foundation **+** the simple, optionless state actions — **Close, Reopen, Mark ready-for-review, Convert-to-draft**. No merge.
- **Slice 2 (future, separate spec):** **Merge** — merge-method choice (merge/squash/rebase, respecting repo-allowed methods), a net-new repo-allowed-methods read, eligibility gating off `MergeReadiness`, and a stronger confirmation. Slice 2 drops its merge box into the **same** Overview panel this slice builds (see UX), so no re-layout.

Rationale: merge carries all the genuinely hard parts (method picker, repo-settings read, eligibility, strong confirm). Landing the foundation in front of the *simplest* real actions exercises the plumbing with shipping behavior (not abstractly), keeps each diff reviewable, and gets the reusable layer #571 needs in sooner.

**Primary driver (positioning).** The lifecycle actions are infrequent (≈once per PR) and Open-in-GitHub already reaches them, so the marginal in-app convenience is modest. The principal justification is the **reusable write-path foundation** #571 (and later slice 2's merge) needs — the actions are the smallest shipping vehicle that exercises it. This is also PRism's **first non-review GitHub write**: it deliberately moves the tool from review-companion toward PR-lifecycle management, and each action is permanent GitHub-API surface (REST state, GraphQL mutations, node-id resolution, 403/branch-protection edges) a single maintainer commits to. That trajectory bet (slice 2 merge next) is accepted, not incidental.

## Resolved design decisions

Each of these was settled in brainstorming; recorded here so the plan and reviewers inherit the rationale (not just the conclusion).

1. **Update model — pending-state-then-reconcile, *not* optimistic-flip-with-rollback.** The issue text asked for optimistic+rollback; we rejected it for *these* actions. They are infrequent (once per PR) so the latency payoff is low, and a PR's state isn't one flag — flipping open→closed changes the glyph, the status label, *which buttons render*, and the readiness badge, so optimistic mutation + clean rollback on a 403 is real bug surface. Instead: the invoked button shows a spinner and disables its siblings; the **pending flag clears on the POST 200** (the in-flight request is done — see "reconcile mechanism" below for why pending does *not* wait on the reload); the authoritative new state then arrives via a dedicated reload (below) and swaps the button set. On failure the spinner clears and a toast explains. The perceived ~300ms–1s gap reads as "working" for a deliberate action. (Optimistic flip may suit #571's high-frequency thread-resolve later; the foundation hook leaves room for it as an opt-in, but slice 1 does not build it.)

2. **Confirmation — per action.**
   - **Reopen / Mark ready-for-review / Convert-to-draft** — non-destructive and reversible; **no confirmation**, just a spinner.
   - **Close** — reversible (Reopen exists), so it does **not** clear the bar for a blocking modal (`design-handoff` / behavioral guidance: *modals only for blocking decisions*). Guard it with an **inline two-step confirm**: the Close control morphs in place to "Close this PR? — Cancel / Confirm close". Reuses the in-place-confirm idiom already in `PrHeader` (the discard affordances). Full interaction contract:
     - **Dismissal:** Cancel, Escape, or a click outside the control all abort and restore the plain Close button. Tab-out does **not** dismiss.
     - **Focus / a11y:** on morph, focus moves to the Cancel button; an `aria-live` region announces the confirm prompt.
     - **Sibling lock:** while the confirm is open, **all other panel buttons are disabled** (same treatment as pending) so a conflicting action (e.g. Convert-to-draft sitting in the same row) can't be invoked mid-confirm. Confirm → transitions to pending (spinner); Cancel/dismiss → restores the full row.
     - **Failure reset:** on **any** POST failure (permission or generic), **both** the pending flag and the confirming state reset to the plain Close button **before** the error toast — otherwise the control could sit in the confirming state and a follow-up misclick re-fires the close, defeating the guard.

3. **Placement — adaptive bottom-sticky "PR actions" panel in the Overview tab; Overview-only.** The PR header (`PrHeader.tsx`) is already dense (action cluster = Refresh + Open-in-GitHub + Review action button; subtitle row carries the expanded readiness badge + chips). Lifecycle actions go in a **sticky panel pinned to the bottom of the Overview scroll area**, matching GitHub's own merge-box-at-bottom convention — improved by being sticky (no scrolling past a long timeline). The panel is **state-aware** (renders only the actions valid for the current state), stays **compact** (single row) for 1–2 buttons, and is the **same container** slice 2's merge box expands into. When **no** lifecycle action applies (merged), the panel is **not rendered** (no empty chrome).
   - **Structural containment (not pixels):** the panel must be a **full-width sticky footer of the Overview tab**, mounted as a sibling of the ~80%-width content column rather than inside it (mounting inside the padded content column would render a narrow, inset bar and fight `position: sticky` against the column's box). This is an information-architecture/DOM-parent decision, fixed here; the exact spacing/tokens/both-theme treatment is the part deferred to the gate.
   - **Required DOM restructure (corrected — round 2 feasibility, verified `OverviewTab.tsx:107`).** `.overviewTab` and `.overviewGrid` are **not** a parent/child wrapper pair today — both class names are applied to the **same single `<div>`** (`<div className={`${styles.overviewTab} ${styles.overviewGrid}`} data-testid="overview-tab">`): `.overviewTab` supplies `flex:1; overflow:auto` (the scroller) and `.overviewGrid` supplies `width: min(80%, …)` (the content column), collapsed onto one node. So "inside the grid" and "direct child of the tab" are currently the *same place*, and the full-width-footer layout this slice fixes **does not exist yet**. The plan must therefore **split that node into an outer `.overviewTab` scroll container wrapping an inner `.overviewGrid` content column**, mount `PrActionsPanel` as a direct child of the **outer** (after the grid), and re-verify that (a) the `data-testid="overview-tab"` lands where existing tests/e2e locators expect it and (b) `position: sticky` behaves against the outer `overflow:auto`. This is unstated structural work, not a one-line append.
   - **Accepted tradeoff:** lifecycle actions are reachable only from Overview, not from Files/Checks/Drafts. This asymmetry is deliberate: review-submit follows you cross-tab (in the header) because it's the end of the Files-reading flow; lifecycle is a "step back and act on the whole PR" decision, and Overview is the PR's home.
   - **Final visual polish (exact spacing/tokens/both-theme treatment) is deferred to the B1 visual gate** — owner will tweak as it renders. This spec fixes *placement, structure, behavior, and state-awareness*, not pixels.

4. **Interface structuring — new `IPrLifecycleWriter`, separate from `IReviewSubmitter`.** Lifecycle actions are not reviews; mixing them muddies `IReviewSubmitter` and forces its several test fakes to implement four irrelevant methods (CS0535 fake-churn). New interface keeps the seam clean.

5. **PAT-scope handling — reactive, not proactive.** The classic `repo` scope PRism's auth validator requires for **classic** tokens (`GitHubAuthValidator.cs`) grants PR write, so a proactive token-capability probe is dead code for those users *and* still can't predict branch-protection 403s. Handle the write failure reactively: classify it (see error handling) and surface a clear, **actionable** message.
   - **Fine-grained PAT caveat (corrected):** `GitHubAuthValidator` also **accepts fine-grained PATs and does not scope-check them (fail-open)**. So a fine-grained token with read-but-not-`Pull requests: write` passes auth, reaches a lifecycle action, and 403s. The reactive copy must therefore be actionable for *both* token types.
   - **Single combined copy, no runtime token-kind switching (corrected — round 2: design + scope-guardian + security).** An earlier draft said the copy was "token-type-aware," which implies the frontend chooses classic-vs-fine-grained text at runtime. **There is no signal for that:** the error DTO carries only a `code`, the snapshot/active-PR state carries no token kind, and the stored-token prefix (`ghp_` vs `github_pat_`) lives only in `TokenStore` on the backend. Rather than add a `tokenKind` field just for copy, the `token-cannot-write` toast is **one message that names both grant paths** — e.g. *"PRism can't change this PR's state. Grant PR-write access: classic PAT → the `repo` scope; fine-grained PAT → 'Pull requests: Read and write'. If you're not a collaborator on this repository, lifecycle actions require collaborator access (a permission grant won't help)."* The non-collaborator clause is included because GitHub returns the **same** *"Resource not accessible by personal access token"* body when the token lacks scope **and** when the user isn't a collaborator — the body alone can't distinguish them, so the copy must cover both rather than dead-end a non-collaborator chasing a scope they already have. ("token-type-aware" as a runtime behavior is dropped; the test asserts the single combined string, not conditional branching.)

6. **Error surface — toast/snackbar via existing `useToast`.** No new error primitive; same `show({ kind: 'error', … })` path the review-submit flow uses, consistent with the app's snackbar-over-banner guidance and #446 direction.

## Architecture

**No GET wire-shape change.** The panel reads PR state from the *existing* fields already on `PrDetailDto.Pr` — `State`, `IsDraft`, `IsClosed`, `IsMerged` (`PRism.Core.Contracts/Pr.cs`). Slice 1 adds **write** endpoints + UI only; it adds no field to any GET DTO. This shrinks blast radius — no new-wire-field sweep of e2e route mocks / FE consumers is required (the trap noted in prior wire-shape work).

The actions ride the established write seam: **endpoint → writer → publish a dedicated bus event (snapshot evict) → frontend reload refetch**, the same shape comment-posts already use (`PrSubmitEndpoints.cs` / the `…PostedBusEvent` → `PrDetailLoader.Invalidate` → SSE → subscriber → `usePrDetail.reload()` chain).

### Reconcile mechanism — a dedicated lifecycle event, NOT `StateChanged` (corrected)

> **Correction from ce-doc-review (feasibility + adversarial, confirmed in code).** An earlier draft claimed an *existing* "`StateChanged` → `usePrDetail.reload()`" path. **That path does not exist.** In `PrDetailView.tsx`, the `StateChanged` subscriber is wired only to `draftSession.refetch` (it refreshes the **draft session**, not the `PrDetailDto` the header glyph/badge and `PrActionsPanel` read). `usePrDetail.reload()` is driven exclusively by **dedicated** subscribers — `useRootCommentPostedSubscriber`, `useSingleCommentPostedSubscriber`, `useDraftSubmittedSubscriber` — each paired with its own `PrDetailLoader.Invalidate` handler. A lifecycle write that published only `StateChanged` would refetch the draft session but **never re-GET the PR detail**, so the header/panel would stay stale and (because `usePrAction` waits for a reload to swap state) the perceived UX would be broken.

The design therefore adds its **own** event + subscriber, mirroring the comment-posted precedent:

- **Backend:** after a successful lifecycle write the endpoint publishes a new bus event **`PrLifecycleChanged(prRef)`** (and may still publish `StateChanged` for draft-session consistency, but the reload does **not** depend on it). `PrDetailLoader` handles `PrLifecycleChanged` with an **unconditional `Invalidate(prRef)`** (exactly like `OnSingleCommentPosted`).
- **SSE wiring is not automatic (round 2 feasibility).** A new bus event reaches the browser only if it is **hand-wired in two files**: `SseChannel.cs` needs a `bus.Subscribe<PrLifecycleChanged>` + a fan-out handler (mirroring `OnSingleCommentPosted → FanoutProjected`) + a `Dispose` entry, and `SseEventProjection.cs` needs a `Project` switch arm + a `…Wire` record. The projection's `default` arm `throw`s for any unprojected event, but `ReviewEventBus.Publish` **isolates and logs** a faulting subscriber rather than propagating — so a missing projection arm means the endpoint still returns 200 and `PrDetailLoader.Invalidate` still evicts, while the **SSE frame is silently dropped** and reconcile silently degrades to the ~5s `usePrAction` fallback (the "immediacy" benefit quietly disappears). The component list and tests below add these touchpoints and an explicit **SSE-frame assertion** (the publish/evict unit tests cannot catch a missing projection arm).
- **Frontend:** a new **`useLifecycleChangedSubscriber`** (mirroring `useSingleCommentPostedSubscriber`) calls **`usePrDetail.reload()`** on the SSE event. **Mount location is `PrDetailView` (the always-mounted PR-detail parent), NOT `OverviewTab`/`PrActionsPanel`** — round 2 design. If it were co-located with the panel, switching to Files/Checks mid-action would unmount the subscriber (and the `usePrAction` fallback timer), the `PrLifecycleChanged` frame would arrive with no listener, and — because `OnActivePrUpdated` doesn't carry the draft flag — the two **draft toggles would be stuck stale until a manual page refresh**. Wiring it in `PrDetailView` (where `useStateChangedSubscriber` and the comment subscribers already live) guarantees it survives tab switches. File location under `frontend/src/hooks` is fine; *mount* location must be `PrDetailView`. This subscriber is part of the reusable foundation — #571 reuses the same event/subscriber shape.
- **Acting-tab reconcile must not flash the transition banner (round 2 adversarial, verified `PrDetailView.tsx:360-378` + `useActivePrUpdates.ts`).** `PrDetailView` derives a *"This PR was closed — Reload"* `BannerTransition` + a one-shot auto-reload from `useActivePrUpdates`, which latches `isClosed`/`isMerged` off the background `pr-updated` SSE. When the user closes the PR from the panel, the backend's own active-PR poll emits `pr-updated{isClosed:true}`; on the **acting** tab, during the window before the lifecycle GET completes, `transitionState` reads `'closed'` and the user who just clicked Close sees a redundant *"PR was closed — Reload"* banner **and** a second `reload()` stacks on the subscriber's. Fix: the lifecycle reconcile must **clear the transition latch for the acting tab** — call `updates.clear()` as part of the reconcile (mirroring `handleReload`/`prRefresh`, which both clear it) so a self-initiated close doesn't read as a background transition. (Reopen is safe — open isn't a done-state; slice 2's Merge will hit the identical path.)
- **Own-tab note:** the comment-posted subscribers process their own tab's events (the `sourceTabId`-match suppression is specific to `StateChanged`/`useStateChangedSubscriber`); the new lifecycle subscriber follows that same "process own event" model, so the acting tab reloads. (See Open Questions for the tabId-threading nuance.)

## Components

### Backend

- **`IPrLifecycleWriter`** (`PRism.Core`) — four methods, each `(PrReference, CancellationToken)` → a typed result (`PrLifecycleResult`: success, or a failure carrying an **error code** so the endpoint can map causes to the right HTTP status + toast):
  - `CloseAsync` / `ReopenAsync` → REST `PATCH /repos/{o}/{r}/pulls/{n}` with body `{ "state": "closed" }` / `{ "state": "open" }`.
  - `MarkReadyForReviewAsync` / `ConvertToDraftAsync` → GraphQL `markPullRequestReadyForReview` / `convertPullRequestToDraft` (REST has **no** draft toggle), keyed by the PR's GraphQL **node id**.
- **`GitHubPrLifecycleWriter`** (`PRism.GitHub`) — implements the interface over the shared `internal static` transport (`GitHubGraphQL.PostAsync`, `GitHubHttp.SendAsync`) and the injected `_readToken` factory, mirroring `GitHubReviewSubmitter`'s construction. (Note — round 2 feasibility: `GitHubReviewSubmitter`'s `PostGraphQLAsync`/`SendGitHubAsync` are *private instance* wrappers, not shareable; the writer copies the same thin wrappers over the shared statics — the established "verbatim twin" pattern.)
  - **Error classification (corrected — do not collapse all 403s).** GitHub returns 403 for **distinct** causes; the codebase already distinguishes them (`GitHubCiFailingDetector.cs` separates fine-grained 403 from rate-limit; `GitHubGraphQLException` reads `extensions.code`). The writer must inspect the **response body / GraphQL `extensions.code`**, not the bare status, and emit:
    - `token-cannot-write` — true permission/scope denial **or non-collaborator** (body ≈ *"Resource not accessible by personal access token"* / GraphQL permission code — GitHub uses the same body for both). Toast = the single combined message of decision 5 (both grant paths + the non-collaborator clause).
    - `repo-rule-blocked` — branch-protection / policy block (body ≈ *"Protected branch"* / *"Validation Failed"*). Toast explains the rule blocked it and **does not** tell the user to change their PAT.
    - `reopen-not-possible` — REST `PATCH {state:"open"}` returns **422** when the head branch/repo no longer exists (common: branches are routinely deleted right after close). Toast names the deleted-head cause; **not** a token/permission message. (Round 2 adversarial.)
    - `plan-unsupported-drafts` — `convertPullRequestToDraft` on a repo whose plan doesn't support draft PRs (tentative; the exact wire shape — structured GraphQL `extensions.code` vs HTTP 422 — is confirmed at plan stage, see Open Questions). Toast names the plan limitation. Defining a tentative code now keeps this off the generic-toast path. (Round 2 design.)
    - Secondary-rate-limit / abuse 403 → route through the existing transport rate-limit path (`ThrowIfRateLimited`), **not** `token-cannot-write`.
    - other failure → generic code.
  - **Server-side failure logging (round 2 security).** Before returning the sanitized structured DTO, log the classified failure **including the truncated GitHub response body** at Warning level, mirroring the existing `s_ownDiscardGitHubFailed` precedent in `PrSubmitEndpoints.cs` (which logs the full message incl. raw body server-side *before* `MapGithubError` strips detail from the client response). This is the first writer that classifies on response body, so the sanitization boundary must be explicit and auditable — and a recurring mis-scoped-token 403 pattern stays observable.
  - **Node-id resolution (corrected — round 2 feasibility).** The two GraphQL mutations need the PR node id. The earlier "thread it from the snapshot" plan is **unavailable**: `PRism.Core.Contracts/Pr.cs` carries **no `node_id`/`NodeId` field** and `PrDetailSnapshot` only wraps that DTO, so the open question "does the snapshot retain node_id?" resolves to **no** (without a new DTO field, which this slice avoids). Instead, **reuse the existing GraphQL resolver pattern** `GitHubReviewSubmitter.Submit.cs:ResolvePullRequestNodeIdAsync` (`repository.pullRequest(number){ id }`) — consistent with the GraphQL transport the mutations already use, rather than introducing a net-new REST `GET …/pulls/{n}` shape with its own 404 handling. This resolve hop **always runs** (one extra GraphQL round-trip per draft toggle) — acceptable for these infrequent actions; it's a standing cost, not a rare fallback.
- **`PrLifecycleEndpoints.cs`** (`PRism.Web/Endpoints`) — one POST per action (matching the existing per-action convention `…/submit`, `…/root-comment/post`, `…/comments/post`):
  - `POST /api/pr/{owner}/{repo}/{number:int}/close`
  - `POST /api/pr/{owner}/{repo}/{number:int}/reopen`
  - `POST /api/pr/{owner}/{repo}/{number:int}/ready-for-review`
  - `POST /api/pr/{owner}/{repo}/{number:int}/convert-to-draft`
  - Each: **auth/subscribed guard + custom-header gate** (see security note) → call the writer → **on success**: publish `PrLifecycleChanged(prRef)` (→ snapshot evict via `PrDetailLoader.Invalidate` + SSE via the `SseChannel`/`SseEventProjection` touchpoints noted under Reconcile) → 200; **on permission/non-collaborator failure**: `{ code: "token-cannot-write" }` → **403**; **branch-protection**: `{ code: "repo-rule-blocked" }` → **403** (distinct code); **reopen-deleted-head**: `{ code: "reopen-not-possible" }` → **422**; **draft-plan-unsupported**: `{ code: "plan-unsupported-drafts" }` (status per confirmed wire shape); other failure → 5xx + generic error DTO.
  - **CSRF parity (security).** The existing submit endpoint requires a non-empty allowlist-conforming `X-PRism-Tab-Id` custom header, which incidentally blocks cross-origin CSRF (browsers can't set custom headers cross-origin without a CORS preflight). `RequireSubscribed` alone does **not** prove the caller is the PRism UI, so a co-browsed malicious page could POST `…/close` for a subscribed PR. The four lifecycle POSTs must carry the **same custom-header requirement** (presence + allowlist match; no head-SHA/drift check needed for lifecycle) to match the write surface they're modeled on.
- **Explicit-invalidation seam (reusable foundation).** Close/reopen/ready/convert **don't move headSha**, and the snapshot cache (`PrDetailLoader._snapshots`) is headSha-keyed — so the key won't auto-swap and a stale snapshot would be served (the same gotcha as comment posts). The `PrLifecycleChanged` handler's unconditional `Invalidate(prRef)` fixes this for all four. #571 reuses this seam.
  - *Why explicit even though `OnActivePrUpdated` exists (corrected):* `OnActivePrUpdated` already evicts on any done-state flip (`doneNow != doneCached`), so **Close *and* Reopen are both caught on the next active-PR poll**. What it does **not** carry is the **draft flag**, so the two draft toggles would never evict via that path. The explicit `PrLifecycleChanged` evict is justified by (a) **immediacy** — all four reconcile now, not next-poll — and (b) covering the draft toggles at all.

### Frontend

- **`usePrAction` hook** (`frontend/src/hooks`, reusable foundation) — pending-state-then-reconcile. Tracks the in-flight action; POSTs via the api client.
  - **Pending clears on the POST 200**, not on reload completion (request-in-flight vs. UI-freshness are separate concerns). The authoritative button set is then swapped by the reload (below) as a subsequent step.
  - **Synchronous re-entrancy guard (round 2: design + adversarial).** React pending state lands on the *next* render, so disabling-via-state alone doesn't stop a second click (or click+Enter) that fires before the re-render — and while REST close/reopen are idempotent, the **GraphQL `markPullRequestReadyForReview`/`convertPullRequestToDraft` mutations are not**: a second call hits an already-ready/already-draft PR and returns a GraphQL error, surfacing an **error toast on an action that actually succeeded** (same trap on a lost-response retry). So `usePrAction` holds a synchronous in-flight **ref** per PR and ignores invokes while one is in flight (before the pending state commits), and the **invoking button itself is disabled** (not just siblings). Additionally, the error classifier treats the "already ready" / "already a draft" GraphQL codes as **benign no-ops** (reconcile, no error toast) rather than failures.
  - **SSE-drop fallback (freshness signal defined — round 2 adversarial).** `usePrAction` does not own `usePrDetail.reload()` (that lives in `PrDetailView`), so "fire the fallback only if the SSE reload hasn't" needs an observable freshness signal. The foundation passes `reload` **plus** `usePrDetail`'s monotonic reload counter (`reloadCounter`) into `usePrAction`: arm a ~5s timer on POST 200, **cancel it when the counter advances** (the SSE reload landed); if it fires, call `reload()` directly. Worst case is one harmless extra GET; the signal keeps the fallback genuinely conditional rather than an unconditional double-reload, and pins the coordination seam #571 inherits.
  - **on failure** clears pending (and the confirm state, per decision 2 — and any open confirm is also cleared by an external `pr`-state change, see PrActionsPanel) and fires an error toast, mapping the error code → copy: `token-cannot-write` → the combined PAT/collaborator copy (decision 5); `repo-rule-blocked` → the policy copy; `reopen-not-possible` → the deleted-head copy; `plan-unsupported-drafts` → the plan-limitation copy; a **subscribe/auth rejection** (the `RequireSubscribed` reject status — confirm code at plan stage) → its **own** "session lost access to this PR — reload" copy, **not** the PAT copy; else generic.
  - Exposes per-action pending flags + invoke functions. #571's resolve/unresolve plugs into the same hook.
- **`useLifecycleChangedSubscriber`** (`frontend/src/hooks` or `PrDetailView` wiring, reusable foundation) — SSE subscriber for `PrLifecycleChanged` → `usePrDetail.reload()`, mirroring `useSingleCommentPostedSubscriber`.
- **`PrActionsPanel`** (`frontend/src/components/PrDetail/OverviewTab`) — the adaptive bottom-sticky panel. Derives visible buttons from `prDetail.pr` (`isDraft` / `isClosed` / `isMerged` / `state`). Wires to `usePrAction`; buttons reflect pending (spinner + invoking-and-sibling buttons disabled) and the Close confirm contract (decision 2). Render rules:
  - `null` when **no action applies** (merged).
  - `null` during **cold load** (`prDetail` null / loading), matching `PrHeader`'s loading-suppression of its action cluster (don't render buttons off undefined `pr` fields).
  - `null` when **`readOnly` is true.** *(Corrected — round 2 design + security: `readOnly` in `OverviewTab` is the **cross-tab write-lock** from `useCrossTabPrPresence` — i.e. "another PRism tab currently holds write on this PR" — **not** a "peer-owned PR" / ownership gate as an earlier draft said.)* Suppressing the panel under that lock keeps lifecycle consistent with how the other writes (`submit`, comment posts) already behave under cross-tab presence. It is **not** an authorization control — real authz is GitHub's PAT enforcement on the server (`RequireSubscribed` + the reactive 403 path); a maintainer acting on someone else's PR is **not** blocked by `readOnly` (it isn't ownership-based), so the feature is not hidden from the triage workflow. Confirm the exact `readOnly` source at plan stage.
  - **Focus on success-driven swap (round 2 design a11y).** When the reload swaps the button set (e.g. Convert-to-draft → Mark-ready), the element under focus is removed. Move focus programmatically to the panel container (or the first button of the new set) so keyboard users don't get dumped to `<body>`.
  - **External state change clears an open confirm (round 2 design).** If an external SSE reload (e.g. a peer closes the PR) changes the computed visible-action set while the inline Close confirm is open, the `isConfirming` state is cleared — same rule as the failure-reset — so the confirm can't sit stranded over an already-closed PR's button set.
  - Mounted by `OverviewTab.tsx` as a **sibling of the content column** (decision 3 — requires the `.overviewTab`/`.overviewGrid` DOM split, since today they're one node).
- **`prLifecycle` client** (`frontend/src/api`) — `closePr` / `reopenPr` / `markReady` / `convertToDraft` fns + the typed error codes (`token-cannot-write`, `repo-rule-blocked`, `reopen-not-possible`, `plan-unsupported-drafts`, subscribe-rejection, generic).

### State → visible actions (the state-awareness matrix)

The panel renders **nothing** in any of these suppression cases first: cold-load, `readOnly`, or merged. Otherwise:

| PR state | Visible actions |
|---|---|
| Open, draft | Mark ready-for-review · Close |
| Open, non-draft | Convert-to-draft · Close *(slice 2 adds Merge here)* |
| Closed (not merged) | Reopen *(can 422 if the head branch was deleted → `reopen-not-possible` toast)* |
| Merged | *(panel not rendered)* |
| Any state, `readOnly` or cold-load | *(panel not rendered)* |

## Data flow

Click (re-entrancy ref blocks a duplicate) → `usePrAction` sets pending (invoking + sibling buttons disable, arms the ~5s freshness timer) → `POST /api/pr/{…}/{action}` (with `X-PRism-Tab-Id`) → writer resolves node id (draft toggles) + hits GitHub → **200** → **`usePrAction` clears pending** (request done). In parallel: endpoint publishes `PrLifecycleChanged` → snapshot evict + SSE projection → `useLifecycleChangedSubscriber` (mounted in `PrDetailView`) → `updates.clear()` (suppress the self-action transition banner) + `usePrDetail.reload()` → reload counter advances (cancels the freshness timer) → fresh `PrDetailDto` → panel re-renders the new button set, focus moves to the new set. *(If the counter hasn't advanced within ~5s, `usePrAction` triggers `reload()` directly.)* **Failure** → error DTO → `usePrAction` clears pending **and** any confirm state + toast (code-mapped).

## Error handling

- **`token-cannot-write` (scope/permission 403 **or** non-collaborator)** → the single combined toast of decision 5: names both grant paths (classic `repo` scope / fine-grained "Pull requests: Read and write") **and** the non-collaborator caveat. No runtime token-kind detection.
- **`repo-rule-blocked` (branch-protection 403)** → toast explaining a repo rule blocked the action; **does not** advise changing the PAT.
- **`reopen-not-possible` (reopen 422, head branch deleted)** → toast naming the deleted-head cause; not a permission message.
- **`plan-unsupported-drafts` (convert-to-draft on a plan without draft PRs)** → toast naming the plan limitation (tentative; wire shape confirmed at plan stage).
- **GraphQL "already ready" / "already a draft"** → treated as a benign no-op (reconcile, **no** error toast) — guards the double-submit / lost-response-retry case.
- **Secondary rate-limit / abuse 403** → routed through the transport rate-limit path, surfaced as a transient/retry message — never `token-cannot-write`.
- **Subscribe/auth rejection** → its own "session lost access — reload" toast, not the PAT copy.
- **TOCTOU** (e.g. Close a PR a peer already closed) → GitHub errors → toast + the `reload()` reconciles the panel to actual state.
- **Close confirm on failure, or an external state change while confirming** → resets to the plain Close button (decision 2 / PrActionsPanel) before any toast.
- **Any other failure** → generic toast, pending cleared. All via `useToast`. Every classified GitHub failure is logged server-side (with truncated body) before the sanitized DTO is returned.

## Acceptance criteria

- [ ] From the Overview tab, on an **open draft** PR I can **Mark ready-for-review** and the header glyph/badge reflect the change without a manual refresh.
- [ ] On an **open non-draft** PR I can **Convert-to-draft** and **Close** (Close behind the inline two-step confirm).
- [ ] On a **closed (not merged)** PR I can **Reopen**.
- [ ] On a **merged** PR, a **`readOnly`** PR, and during **cold load**, the panel is **not shown**.
- [ ] A token lacking PR-write permission produces a single clear toast naming **both** grant paths (classic `repo` / fine-grained "Pull requests: Read and write") and the non-collaborator caveat; a **branch-protection** block produces a distinct toast that does **not** advise changing the PAT; reopening a PR whose head branch was deleted produces a distinct deleted-head toast.
- [ ] Each action shows a pending spinner and disables the **invoking and sibling** buttons while in flight; a double-click does **not** fire a second action or surface a spurious error; **pending clears on the POST response**; state reconciles via the lifecycle SSE/reload (with a ~5s freshness-gated fallback) and the acting user does **not** see a redundant "PR was closed — Reload" banner for their own action; on failure the spinner **and** any Close-confirm state clear and a toast explains.
- [ ] The inline Close confirm is keyboard-operable (Escape cancels, focus moves to Cancel, prompt announced); after a successful action that swaps the button set, focus moves to the new set (not `<body>`).
- [ ] Lifecycle POSTs require the `X-PRism-Tab-Id` custom header (CSRF parity with submit).
- [ ] No change to any GET DTO wire shape.
