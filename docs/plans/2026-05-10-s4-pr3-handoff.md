## Carry-forward from PR2 (load-bearing)

**Two correctness fixes were authored after PR2 merged and were NOT included in the merge.** They live on side branch `prep/pr3-pr2-followups` (commit `191a3336`) — first thing to do in PR3 is fold them in. Both are real bugs PR3's endpoint will hit if not landed.

### Fix 1 — `MakeStale` preserves location metadata for line-anchored Stale outcomes

**Why it matters for PR3:** PR3's `POST /reload` apply-result step writes the pipeline's `ReconciliationResult` back into `state.Reviews.Sessions[refKey]`. The current `MakeStale` zeroes out `ResolvedFilePath` and `ResolvedLineNumber` for ALL Stale outcomes — but spec § 3.2 says these are null only for PR-root drafts. If the apply step takes a line-anchored draft's null `ResolvedFilePath` literally, the persistent draft loses its location and the reconciliation panel can't render the row for the user to re-anchor.

**The fix on `prep/pr3-pr2-followups`:** `MakeStale` takes `resolvedFilePath` and `resolvedLineNumber` as explicit parameters; every Stale call site passes the right value:
- Null-anchor guard → `draft.FilePath`, `draft.LineNumber`.
- File deleted via `deletedPaths` → `draft.FilePath`, `draft.LineNumber`.
- File doesn't exist at `newHeadSha` → `fileResult.ResolvedPath` (post-rename location), `draft.LineNumber`.
- Per-draft exception isolation → `draft.FilePath`, `draft.LineNumber`.

Three new tests pin the contract: `DeleteTests` asserts `FilePath` + `LineNumber` are preserved; `PipelineGuardTests.NullAnchorGuard_PreservesFilePathAndLineNumber`; `PipelineGuardTests.SourceThrowsThenStale_PreservesFilePathAndLineNumber`.

### Fix 2 — `SplitLines` drops trailing-newline phantom line

**Why it matters for PR3:** PR3 will reconcile against real GitHub file content fetched via `IReviewService.GetFileContentAsync`. Real files almost always end with `\n` (POSIX text-file convention). The current `SplitLines` retains a trailing empty element on `\n`-terminated content — a 3-line file becomes 4 entries. An empty-anchored draft (degenerate but constructible) would whitespace-equiv-match the phantom line and produce a spurious end-of-file match.

**The fix on `prep/pr3-pr2-followups`:** `SplitLines` drops exactly one trailing empty element when content ends with `\n`. New test `WhitespaceTests.TrailingNewline_DoesNotProduceSpuriousMatch_AtPhantomLine` pins the contract.
